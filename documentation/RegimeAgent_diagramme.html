Si le RegimeAgent doit servir

Aujourd'hui le RegimeAgent est un module de post-traitement cosmétique : il n'améliore aucune prévision, il assouplit le critère de réussite. Trois voies utiles, déjà dans le brief, remplaceraient l'overlay au lieu de l'élargir :

brief — point 2

Ventiler les métriques par régime

Déjà prévu via volatility_regimes dans honest_eval — le HMM/BOCPD du RegimeAgent ferait mieux que la segmentation actuelle.

brief — point 4

Feature exogène des modèles

stress_score injecté en amont du training (étape 3.1) plutôt qu'en post-traitement à l'étape 5.

brief — point 4

Comparaison de volatilité

Confronter la vol implicite du RegimeAgent (GARCH interne) au GARCH du benchmark sur les cibles de volatilité — plutôt qu'élargir des IC pour faire passer des verdicts.

Note sur le lag / déphasage (ARIMA, SARIMA, LSTM)

Les 3 modèles font du walk-forward : à chaque pas, la vraie valeur réalisée est réinjectée avant de prédire le pas suivant. Directional accuracy mesurée ≈ 52–57 % (quasi pile-ou-face) — le prix réagit comme une marche aléatoire quasi pure. Le prédicteur optimal au sens MSE converge alors vers « prix futur proche du dernier prix connu ».

Ce n'est pas un bug de code : c'est la signature d'un actif proche de l'efficience — cf. brief, Point 1 (évaluer les variations, pas les niveaux).

3.1 (training) et 3.2 (calibration régime) tournent en parallèle sur le même split — aucune dépendance entre elles, hormis l'overlay en aval (étape 5) qui ne dépend que de 3.2.
